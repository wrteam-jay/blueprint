# User Stories

## Spec creation

**US-1.** As a Spec Author, I want to build a blueprint from scratch through guided conversation, so that I can capture a planned or new system without having to know the blueprint format upfront.

**US-2.** As a Spec Author, I want to extract a blueprint from an existing undocumented system, so that tribal knowledge is captured before it is lost to team turnover or memory decay.

**US-3.** As a Spec Author, I want to generate an empty blueprint directory with placeholder files, so that I have the correct structure before starting to fill in content.

## Spec quality

**US-4.** As a Spec Owner, I want a multi-perspective review panel to debate my blueprint's quality, so that blind spots from any single role are surfaced before the team relies on the document.

**US-5.** As a Spec Owner, I want a systematic checklist audit of my blueprint, so that I get a structured findings report without the overhead of a full panel debate.

**US-6.** As a Blueprint Reader, I want every term used in the blueprint to be defined once and used consistently, so that the document means the same thing to everyone regardless of their role.

## Spec evolution

**US-7.** As a Spec Author, I want to evaluate a proposed system change through panel debate before implementing it, so that the team considers multiple perspectives and the blueprint reflects deliberate decisions.

**US-8.** As a Spec Author, I want to incrementally update a blueprint after the system has changed, so that the blueprint stays current without requiring a full rewrite.

**US-9.** As a Spec Owner, I want decisions recorded with rationale (not just outcomes), so that future team members understand why choices were made and don't re-debate settled questions.

## Cross-role access

**US-10.** As a Blueprint Reader (developer), I want scenarios that describe end-to-end flows with preconditions and outcomes, so that I know exactly what to build without guessing at edge cases.

**US-11.** As a Blueprint Reader (QA/tester), I want error paths documented alongside happy paths, so that I can write test cases covering failure modes without reverse-engineering them from code.

**US-12.** As a Blueprint Reader (designer), I want actor definitions and user stories grounded in evidence, so that I design for actual user needs rather than assumptions.

**US-13.** As a Blueprint Reader (product manager), I want a behaviour-level spec free of implementation details, so that I can make prioritisation and scoping decisions without needing to understand the codebase.

**US-14.** As a Blueprint Reader (business stakeholder), I want business rules sourced to their origin (policy, regulation, or decision), so that I know which rules I can change and which are externally imposed.
